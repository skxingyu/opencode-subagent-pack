# 任务委托策略

当遇到符合条件的情况时，使用 Task 工具自动派发给对应子 agent，由子 agent 独立完成工作并返回结果。这可以避免污染主上下文并提高效率。

## 委托给 @researcher

以下情况应当委托给 researcher：
- 需要 10 条及以上搜索查询才能完成的调研（如"全面对比一下XX和YY的架构差异"）
- 明显的不确定场景，一波搜索大概率不够（如"这个技术方案市面上有哪些主流实现"）
- 需要深入理解代码库架构或业务流程
- 问题复杂、预计耗时较长或会占用大量上下文

以下情况**不需要**委托 researcher：
- 简单的问答或单步查询（如查一个函数定义、看一个文件内容）
- 10 条搜索以内的信息查证（主 agent 自己查效率更高）
- 只需主 agent 已有知识即可回答的问题

**特殊条件**：如果用户明确要求调用 researcher（如"你用researcher查一下..."）或者明确说"需要高效率"，此时也应当委托 researcher。

researcher 完成调研后返回结构化报告，你阅读后继续后续工作。

## 委托给 @reviewer

以下情况应当委托给 reviewer：
- 需要对代码进行系统性审查（如"审查一下这个PR"）
- 需要检查代码安全、性能和最佳实践
- 需要检测 Bug、边缘情况和逻辑漏洞
- 需要运行 linter 或测试辅助检查

**自动触发**：当你完成代码编写或修改后，如果涉及的新增/修改代码总量超过 200 行，自动调用 reviewer 进行审查。200 行以下的简单改动无需触发。

reviewer 返回审查报告和修改建议，你根据报告决定是否执行修改。

## 不需要委托的情况

以下情况不需要委托：
- 简单的单步问题（如"这个函数是干什么的"）
- 用户明确要求你直接处理
- 只需返回已有知识的问答
